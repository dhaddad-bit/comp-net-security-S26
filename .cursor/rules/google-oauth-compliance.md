# Social Scheduler Google OAuth Compliance Rules

You are assisting on Social Scheduler as a security-first developer.

## Purpose

Apply OWASP ASVS-aligned guidance to all OAuth and Google Calendar integration work.
Prioritize security controls that reduce CASA-style review risk while preserving current app behavior.

## Scope

These rules apply whenever a task touches:

* Google OAuth login flow
* OAuth callback handlers
* token storage/refresh logic
* consent scopes and Google API calls
* frontend auth redirects and session handling related to Google OAuth

## ASVS focus

Primary sections:

* V10: Malicious Code and Data Protection for OAuth identity controls
* V3: Session management and cookie protection
* V2: Input and data validation on callback/query/body values
* V11: API/network controls where OAuth endpoints are externally exposed

## Non-negotiable OAuth controls

### 1) Scope allowlist only (least privilege)

Do not add broad wildcard scopes.
Only allow explicitly approved scopes for this app:

* `https://www.googleapis.com/auth/calendar.readonly`
* `https://www.googleapis.com/auth/userinfo.email`
* `https://www.googleapis.com/auth/userinfo.profile`

If a new scope is requested:

* explain exact business need
* list security impact
* provide rollback/removal plan
* require explicit user approval before merge

### 2) State parameter verification is required

OAuth callback logic must cryptographically validate `state` against session-stored state.
Do not ship with state-check code commented out or bypassed.

Required behavior:

* generate high-entropy state per auth attempt
* bind state to session
* reject callback on mismatch with 403-style response and audit log
* clear one-time state value after successful verification

### 3) Token handling at rest

Do not store refresh tokens in plaintext in persistent storage.
Use authenticated encryption at rest (AES-256-GCM) and key from environment configuration.

Required design:

* encrypt before DB write
* decrypt only when needed for token refresh/API calls
* store IV/nonce and auth tag alongside ciphertext
* fail closed when key material is missing
* never print token values in logs

### 4) Redirect URI and origin exactness

OAuth client configuration and app runtime values must exactly match:

* `GOOGLE_REDIRECT_URI`
* authorized redirect URIs in Google Cloud
* authorized JS origins in Google Cloud

Always verify by inspecting the redirect `Location` generated by `/auth/google`.

## Session and cookie rules for OAuth

* Session cookies must be `HttpOnly`.
* Session cookies must be `Secure` in production.
* `SameSite=Lax` is acceptable and preferred for OAuth redirect compatibility.
* Do not set `SameSite=None` unless there is a documented cross-site requirement and explicit approval.

## Data governance / lifecycle readiness

If code creates provider-owned calendar resources, ensure data ownership fields are explicit and auditable.
For Google lifecycle readiness tasks (for example `dataOwner` support), treat them as compliance-gated changes:

* include migration notes
* include failure behavior for missing ownership metadata
* include cleanup behavior that avoids orphaned resources

## Failure triage map

When OAuth breaks, debug in this order:

1. `invalid_client`:
   * verify runtime client ID/secret loaded in backend process
   * verify OAuth client exists and is a Web Application
2. `redirect_uri_mismatch`:
   * verify callback URI exact string match in Google Cloud and env
3. `invalid_scope`:
   * verify requested scopes against approved allowlist
4. callback 403 / CSRF:
   * verify state generation, persistence, and comparison path

## Logging and secrets

Never log:

* client secret
* access token
* refresh token
* raw authorization codes

When handling incidents, rotate exposed OAuth secrets immediately and update production env.

## Required verification for OAuth changes

A task touching OAuth is not complete unless all are true or explicitly blocked:

* backend starts with required OAuth env vars
* `/auth/google` redirect contains expected `client_id` and `redirect_uri`
* callback path validates state and rejects mismatch
* no plaintext refresh token persistence in changed paths
* no secret leakage in added logs/tests

## Preferred completion summary

End OAuth-related work with:

* changed files
* controls added/verified (state, scope, token storage, cookie flags)
* what was tested
* remaining risk and next smallest hardening slice
